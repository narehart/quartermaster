# 2026-07 cost A/B benchmark

A preregistered A/B test of Quartermaster's cost claims, run 2026-07-13/14.
This is the receipt: what was measured, what it found, and what it doesn't
tell you. The headline result went **against** this project's own
hypothesis, and is reported here in full rather than quietly walked back.

Referenced from [README.md](../../README.md), [AGENTS.md](../../AGENTS.md),
and [ADR 0001](../adr/0001-cost-tiered-delegation.md).

## tl;dr

- **Governance claim holds.** Orchestrator hard-denial (no Edit/Write/Bash,
  no MCP tools on the main thread) held in every containerized run, both
  arms, both experiments.
- **Expensive-model token reduction holds.** At M=opus, Quartermaster cut
  opus-thread tokens by **82.7%** pooled (57%–90% per task) by moving
  sub-agent work to sonnet/haiku — exactly the mechanism the architecture is
  built on.
- **Net per-task cost saving does NOT hold.** At M=sonnet (pilot, n=60), the
  cost-per-solved-task difference was statistically **null** (95% CI
  straddled zero). At M=opus (main experiment, preregistered, n=54 valid),
  Quartermaster cost **1.39x more** per solved task than stock Claude Code
  (95% CI [1.06x, 1.84x]) — a reversal of the hypothesis. Mechanism: total
  token volume roughly doubled under delegation (more round-trips, re-sent
  context per sub-agent call), and that overhead exceeded the dollar savings
  from moving tokens to cheaper per-token tiers.
- **Also slower**, both experiments: ~1.27x median exec time at M=sonnet,
  ~1.92x at M=opus (setup time excluded from both).
- **Untested:** long interactive-session economics — cross-turn prompt
  caching, context reuse across many turns in one session. Both experiments
  were one-shot, cold-start trials (fresh container, no session reuse across
  reps). It is plausible real interactive use looks different; it is not
  demonstrated here either way.

## Why this benchmark exists

Quartermaster's tagline and several docs described it as a way to cut Claude
Code costs. That claim had never been measured against stock Claude Code —
it was an architectural inference ("execution tokens billed at a cheaper
model must cost less overall"), not a benchmarked result. This experiment
tests that inference directly, preregistering the hypothesis and analysis
plan before running the expensive (opus) arm, so the result can't be
rationalized after the fact regardless of which way it comes out.

## Design

**Harness:** [Harbor](https://github.com/laude-institute/harbor) running
Terminal-Bench 2.0 (TB2.0) tasks, each in its own Docker environment; agent
= Harbor's installed `ClaudeCode` agent, driving the actual `claude` CLI
inside the container (not a mock).

**Arms:**
- **Arm A (stock):** vanilla Claude Code, no Quartermaster.
- **Arm B (Quartermaster):** Claude Code run as `--agent orchestrator` with
  Quartermaster installed from the plugin source; sub-agent tiers as
  shipped (`scout`/`mechanic` pinned to haiku, `builder` pinned to sonnet;
  main thread pinned to the experiment's `M`). The orchestrator holds no
  Edit/Write/Bash, so all execution routes through one of the three tiers —
  the same hard-denial invariant this repo enforces (see
  [ADR 0002](../adr/0002-orchestrator-hard-denial.md)).

Both arms: same task pool per experiment, same per-run budget cap, same
concurrency settings.

**Two experiments, two main-thread models (`M`):**

| | model (M) | n (nominal) | status |
|---|---|---|---|
| Pilot | sonnet | 60 (10 tasks x 2 arms x 3 reps) | informal, not preregistered — motivated the follow-up |
| Main experiment | opus | 60 (6 tasks x 2 arms x 5 reps), 54 valid after 6 errors | preregistered before wave 1 launched |

The opus experiment used a smaller, deliberately harder task pool (the 6
longest-running tasks with at least one Arm A solve in prior pilot/profiling
data — see Task selection below) to concentrate spend on the tasks where the
tiering mechanism has the most token volume to act on.

## Preregistration & rules (opus experiment)

Locked in `PREREG_OPUS.md` before any opus-arm data existed beyond a single
technical smoke pair. Key rules:

- **Task selection, deterministic, no cherry-picking:** rank all tasks with
  a completed Arm A baseline by mean true prompt-side tokens; exclude any
  task with a broken environment or **zero** Arm A solves across all prior
  data (cost-per-solved needs a nonzero solve count to be defined); take the
  top 6 remaining by token volume. See "Task selection bias" below — this
  rule is not neutral.
- **Primary metric:** cost-per-solved (`sum(true_cost_usd over all valid
  runs, incl. failures) / n_solved`), task-level then pooled, with a
  bootstrap CI (10,000 resamples, resampling runs within each arm
  independently, paired per iteration).
- **Decision rule, stated before wave 1:** "If the bootstrap CI on the
  primary metric straddles zero, the result is null — regardless of which
  arm's point estimate looks lower. No post-hoc rationalization of a point
  estimate that isn't backed by a CI excluding zero."
- **Instrumentation:** token/cost figures are read from the *true* per-trial
  usage (`extract_true_metrics.py`, summing every `modelUsage[*]` entry off
  the final `{"type":"result",...}` line of the agent's own transcript),
  never Harbor's own `trajectory.json`/`result.json` fields, which are
  main-thread-only and undercount actual usage by up to 45x whenever
  sub-agents run.
- **Errored trials** (harness/agent exceptions — timeouts, crashes) are
  excluded from every cost/pass computation, regardless of any stray
  populated `reward`/cost value on that row; `status` is authoritative.

## Results: pilot (M=sonnet, informal)

n=60 valid runs (10 tasks x 2 arms x 3 reps), plus 1 known Arm A error
(hit the per-run budget cap).

| | Arm A (stock) | Arm B (Quartermaster) |
|---|---|---|
| Pass rate | 17/30 = 56.7% | 20/30 = 66.7% |
| Cost per solved | $1.0230 | $0.8594 |
| Median exec wall-clock | 237.59s | 302.81s (1.27x slower) |

Point estimates favor Quartermaster (16% lower cost-per-solved, higher pass
rate), but **none of it clears statistical significance at n=30/arm**:

- Bootstrap 95% CI on cost-per-solved difference (A − B): **[$-0.37,
  $0.83]** — straddles zero. Verdict: **null**, not "Quartermaster wins."
- Fisher's exact test on pass rate: p = 0.60 (not significant).
- Mann-Whitney U on per-run cost: p = 0.78 (not significant).
- Power analysis: the observed n=30/arm is far under what either effect
  would need for 80% power (≈370/arm for the pass-rate effect, ≈79,369/arm
  for the cost effect at the observed effect size) — this pilot could not
  have reliably detected an effect this size even if a true difference
  exists.

This null (not "small win") result is what motivated the preregistered
follow-up at M=opus, where the architecture's own hypothesis predicts the
tiering mechanism should matter *more* (a bigger price gap between the
main-thread model and the sub-agent tiers).

## Results: main experiment (M=opus, preregistered)

n=54 valid runs (of 60 nominal; 6 errored — 2 Arm A, 4 Arm B, all timeouts
or a process crash, none hit the $15 per-run budget cap).

| | Arm A (stock) | Arm B (Quartermaster) |
|---|---|---|
| Pass rate (valid runs) | 28/28 = 100% | 25/26 = 96.2% |
| Cost per solved | $0.7573 | $1.0507 |
| Median exec wall-clock (setup excluded) | 231.87s | 445.97s (1.92x slower) |

**Primary result:** bootstrap 95% CI on the cost-per-solved difference (A −
B) = **[$-0.55, $-0.05]** — excludes zero. Per the prereg's own locked
decision rule, this is **not null**. But it runs opposite the hypothesis:
Arm B (Quartermaster) cost **1.39x** Arm A's cost per solve (bootstrap CI on
the ratio: **[1.06x, 1.84x]**), with only a 0.8% bootstrap probability that
Arm B was actually cheaper. Pass rates were statistically indistinguishable
(Fisher p = 0.48).

Per-task, the story is heterogeneous, not a uniform tax: Quartermaster was
actually cheaper on 3 of 6 tasks (adaptive-rejection-sampler 0.93x,
break-filter-js-from-html 0.94x, fix-ocaml-gc 0.72x) and substantially more
expensive on the other 3 (bn-fit-modify 3.03x, build-pmars 2.55x,
cobol-modernization 1.59x) — the pooled reversal is driven by those three.

## Mechanism decomposition (the "why")

This is the important part — it is not a "tiering doesn't work" story.

| arm | tier | tokens (prompt+output) | dollars |
|---|---|---|---|
| A (stock) | opus | 14,068,271 | $21.21 |
| B (Quartermaster) | opus | 2,430,871 | $12.12 |
| B (Quartermaster) | sonnet | 14,022,786 | $11.04 |
| B (Quartermaster) | haiku | 9,485,624 | $3.10 |

The tiering mechanism works exactly as designed: **Arm B cut opus-thread
tokens by 82.7% pooled** (57%–90% per task) by moving sub-agent work to
sonnet/haiku. But that opus-token cut only saved about $9 in opus spend
pooled across all valid runs ($21.21 → $12.12), while the sonnet tier alone
added back $11.04 and haiku added $3.10 — a net **+$14.14** in new tiered
spend against **$9.08** in opus savings.

The reason isn't that sonnet/haiku are mispriced relative to opus — it's
that **Arm B's total token volume roughly doubled** versus Arm A's (Arm A:
14.07M tokens, all opus; Arm B: ~25.9M tokens across three tiers). That's
consistent with delegation overhead — multiple scout/mechanic/builder
round-trips, context re-transmitted fresh to each sub-agent call — outweighing
the per-token price cut on the tokens that did move down-tier. Wall-clock
backs this up: 1.92x Arm A's median exec time even with setup excluded —
more hops, not fewer.

## Experiment 1: precision briefs (2026-07-14)

The mechanism decomposition above pointed at delegation overhead — repeated
sub-agent discovery turns and re-sent context — as the dominant cost driver
(`OVERHEAD_DECOMP.md`'s estimate: sub-agent cache-read growth alone accounted
for ~189% of the net A/B gap). This experiment tests a direct intervention on
that lever: a "precision-brief" delegation contract (`templates/
orchestrator.md` requires front-loading exact paths, inlined snippets under
~50 lines, and verbatim commands into every brief; `templates/{scout,
mechanic,builder}.md` add a matching "Stay inside the brief" section — no
exploration beyond the brief, a `NEED_INFO` escape hatch for a genuinely
missing precondition instead of open-ended searching).

**Design:** one arm (Arm B-briefs, Quartermaster + this template change) vs
the existing Arm B baseline from the main opus experiment above — same 6-task
solvable-grind pool, same M=opus main thread, n=30 nominal per arm (28/26
valid after errors). Preregistered in `PREREG_EXP1.md` before any Arm B-briefs
data existed; per that prereg, this experiment makes no new Arm A (stock) run
and no new hypothesis test against Arm A. Spend: $23.19.

**Primary metric: NULL, per the prereg's own locked decision rule.**
Cost-per-solved: briefs $0.8589 vs baseline $1.0507 (point-estimate ratio
0.82x). Bootstrap 95% CI on the difference (briefs − baseline):
**[-$0.448, +$0.074]** — straddles zero. P(briefs cheaper) across bootstrap
draws: 92.5%, but per "if the bootstrap CI on cost-per-solved straddles zero,
the result is null, regardless of which point estimate looks lower," this
does **not** clear the bar to claim a cost reduction. n=28/26 valid runs is
too small relative to rep-to-rep cost variance for this specific comparison
to reach significance.

**Mechanism metrics are decisive, even though the primary cost result is
null — this is the informative part of the experiment.** Every mechanism
metric moved in the predicted direction, by a large margin, on 6/6 tasks:

| metric | briefs median/trial | baseline median/trial |
|---|---|---|
| sub-agent assistant turns | 15.0 | 31.0 |
| sub-agent cache-read tokens | 180,709 | 456,568 |

Median sub-agent turns nearly halved; median cache-read tokens dropped by
more than half. Wall-clock corroborates: briefs' median exec time is **0.78x**
baseline's (349.93s vs 445.97s), consistent with fewer discovery turns per
spawn. Pass rate held flat: briefs 27/28 = 96.4% vs baseline 25/26 = 96.2%
(Fisher p = 1.00, not significant — no regression). The `NEED_INFO` escape
hatch fired once in 28 valid briefs trials (fix-ocaml-gc/wave1, a mechanic
correctly stopping when a brief's git-repo assumption didn't hold) vs 0 in
baseline — a real, appropriate use, not a new source of round-trip overhead.

**Context vs stock claude-code (point estimate only, not a new hypothesis
test):** ratio-vs-Arm-A cost-per-solved moved from the baseline's 1.39x
(Quartermaster ~39% more expensive than stock per solved task) to briefs'
**1.13x** point estimate — directionally consistent with closing the gap
toward parity, though this is not backed by its own bootstrap CI against Arm
A and should not be read as a statistically-confirmed narrowing.

**Caveats:**
- **Tree-version drift.** Arm B baseline ran against the 0.6.2-era mainline
  tree; Arm B-briefs ran against 0.7.0+ (mainline plus the precision-briefs
  template edits). Any mainline changes between those two points beyond the
  template diff are confounded with the intervention and were not
  independently verified commit-by-commit.
- **Small n for a null-vs-not-null call.** 28/26 valid runs is enough to see
  a large, consistent mechanism-level effect (every per-task turn/cache-read
  comparison moved the same direction) but not enough for the pooled
  cost-per-solved CI to exclude zero.
- **Shared task selection.** Same 6-task "solvable-grind" pool as the main
  opus experiment (see task-selection-bias caveat below) — findings here,
  including the mechanism improvement, are conditional on that same regime,
  not a claim that generalizes to the harder zero-solve tail no arm has ever
  solved.

**Shipping rationale:** the primary cost claim is not evidenced (null), and
this document makes no cost-savings claim from this experiment. What is
evidenced is that the precision-brief contract measurably reduces the
delegation-overhead mechanism this whole benchmark series identified as the
cost driver — fewer sub-agent turns, less re-discovered context, faster
wall-clock — with no pass-rate regression and only one appropriately-used
escape-hatch invocation. That is the basis for shipping the template change,
not a cost-savings number.

## Caveats and threats to validity

- **Task selection bias ("solvable grinds").** The opus experiment's prereg
  rule excludes any task with zero Arm A solves in prior data. This
  systematically drops the hardest, longest, zero-solve tail (regex-chess,
  feal-linear-cryptanalysis, circuit-fibsqrt, overfull-hbox,
  build-cython-ext) — exactly the tasks where the tiering mechanism has the
  most raw token volume to act on, since an agent that's failing tends to
  accumulate more sub-agent/tool tokens while struggling. **The reported
  reversal is conditional on "solvable at all by stock Claude Code within
  budget"** and does not generalize to that harder, zero-solve tail; it is
  plausible (not proven) Quartermaster's economics look different there.
- **One-shot, cold-start setting.** Neither experiment reused a session or
  warmed a prompt cache across reps/waves — every trial started from a
  fresh container. Repeated real-world use with cross-session caching,
  context reuse, and warm sub-agent conventions could change the arithmetic
  behind the sonnet/haiku token volume currently driving the reversal. This
  is explicitly untested, not ruled out.
- **Errored-run asymmetry (opus experiment).** 6 of 60 nominal runs errored
  (timeouts/crash, none hit the budget cap) — 2 Arm A, 4 Arm B — concentrated
  on the two highest-token-volume tasks in the pool. Timeouts right-censor:
  a killed run had not yet produced a final result and would plausibly have
  accumulated *more* cost had it continued. Excluding these plausibly
  **understates** Arm B's true cost-per-solved further — the reported
  reversal is, if anything, a conservative estimate of Arm B's cost
  disadvantage, not an inflated one.
- **Modest n.** n=54 valid runs (opus) / n=60 valid runs (sonnet pilot) —
  real, not noise (opus: rep-to-rep-vs-between-arm noise ratio 0.76:1, much
  healthier than the pilot's 38:1), but not a large-n result either.
- **Budget-cap censoring (pilot only).** One Arm A run hit the pilot's $2
  cap and crashed; this plausibly understates Arm A's true cost on its
  hardest trials, biasing the pilot comparison *in favor of* Arm A (i.e.
  against Quartermaster's pilot point estimate looking even better than it
  already does) — doesn't affect the opus experiment, which used a $15 cap
  no run came close to.

## What this does and doesn't tell you

**Evidenced, keep claiming:**
- Least-privilege tool governance — the orchestrator is physically unable
  to hold Edit/Write/Bash/MCP tools, verified holding in every containerized
  run of both experiments.
- Enforced delegation — all execution routes through a tiered sub-agent by
  construction, not by policy alone.
- Expensive-model token-share reduction — 82.7% pooled opus-token cut,
  measured, both experiments' full mechanism decomposition backs this up.

**Not evidenced, stop claiming:**
- "Quartermaster reduces your Claude Code costs" as a general, per-task
  claim. Under one-shot cold-start conditions, delegation overhead roughly
  doubles total token volume, and on the tested opus configuration that
  overhead exceeded the tiering savings — a net 1.39x cost *increase* per
  solved task, not a decrease. At the sonnet main-thread model the same
  comparison was statistically null.
- Any specific cost-savings percentage as a blanket number — the measured
  effect is heterogeneous by task and reverses sign depending on the
  main-thread model.

**Still open:**
- Long-lived interactive session economics (this was never tested — no
  cross-turn caching, no session reuse).
- Whether the mechanism's economics differ on the harder, zero-Arm-A-solve
  tail excluded by the opus prereg's task-selection rule.

## How to re-run

Runner scripts, task pool config, and raw per-trial data live in
`~/qm-bench-spike` (not part of this repo — a separate spike checkout with
its own `harbor`/TB2.0 dependency, `.env` with API credentials, and prebuilt
Docker toolchain mount).

- Pilot (sonnet, informal): `run_pilot.sh` (`gen_config.py`/`regen_config.py`
  generate per-wave Harbor job configs; `analyze.py` produces `ANALYSIS.md`;
  `stats_test.py` produces `STATS.md`). Total spend: ~$35 across 60 runs.
- Main experiment (opus, preregistered): `run_opus_waves.sh`
  (`gen_opus_wave_config.py` generates each wave's job config;
  `analyze_opus.py` produces `OPUS_ANALYSIS.md`). Waves check cumulative
  spend and abort past $250; actual total spend for the full 60-run run was
  **~$50**. Preregistration is locked in `PREREG_OPUS.md` — re-running
  against the same task pool and rules reproduces the same analysis
  pipeline exactly; a new task pool or model would need a new prereg.
- Both use `extract_true_metrics.py` to pull true per-trial token/cost
  figures from each trial's `agent/claude-code.txt` transcript (never
  Harbor's own main-thread-only `result.json`/`trajectory.json` fields,
  which undercount whenever sub-agents run).

Source data: `pilot_results.csv` (60 rows), `opus_results.csv` (60 rows,
including the 6 errored). Both are the raw input to their respective
`analyze*.py` scripts — every number in this document traces back to one of
these two CSVs.
